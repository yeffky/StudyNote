# 锁

解决数据并发访问的一致性、有效性

## 一、全局锁

### 1. 介绍

全局锁就是对整个数据库加锁，加锁后实例处于只读状态。后续的DML、DDL语句，已经更新操作的事务提交语句将被阻塞

### 2. 一致性数据备份

```mysql
flush tables with read lock;
mysqldump -u[user] -p[password] db_name > path/db_name.sql
unlock tables;
```

### 3.特点

* 备份期间，无法写入
* 备份期间从库不能执行主库同步过来的二进制日志，导致主从延迟
* --single-transaction可以完成不加锁的一致性数据备份(mysqldump --single-transaction)

## 二、表级锁

### 1. 表锁

* 表共享读锁
* 表独占写锁

```mysql
lock tables tb_name read/write：加锁
unlock tables/断开客户端连接：释放
```

* 读锁：当前用户和其他用户均不可写
* 写锁：仅当前用户可以读写

### 2.元数据锁

系统控制，访问表时自动添加。避免DML与DDL冲突

* select：shared_read（共享读）
* insert/update/delete/select:shared_write（共享写）（事务不冲突）
* alter table:exclusive（当事务中在进行读写操作时将被阻塞）
* 查询元数据锁：

```mysql
select object_type, object_schema, object_name, lock_type, lock_duration from performance_schema.metadata_locks;
```

### 3.意向锁

避免DML在执行时行锁与表锁的冲突，使得表锁不用检查每行数据是否加锁，减少表锁的检查

上行锁时加上了意向锁，若意向锁和表锁兼容则线程可以操作，不兼容则阻塞线程，直到行锁和意向锁释放

#### (1) 意向锁类型

* 意向共享锁（IS）：由select...lock in share mode添加，和表锁共享锁兼容，和表锁排它锁互斥
* 意向排它锁（IX）：由insert、update、delete、select...for update添加，与表锁都互斥
* 意向锁之间不互斥

#### (2) 查看意向锁

```mysql
select object_schema, object_name, index_name, lock_type, lock_mode, lock_data from performance_schema.data_locks;
```

### 4. 行级锁

粒度最小，冲突概率最低，并发度最高，应用于innoDB

行锁通过对索引上的索引项加锁实现，而非对记录加锁，如果检索数据不涉及索引，则升级为表锁

#### (1) 行锁

锁定单个行记录的锁，防止其他事物进行update和delete，在RC和RR隔离级别下都支持

##### ① 共享锁：允许一个事务读一行，阻止其他事务获得相同数据集的排它锁，只有共享锁和共享锁兼容：select lock in share mode

##### ② 排它锁：允许获取排它锁的事务更新数据，阻止其他事务获得相同数据集的共享锁和排它锁：insert、update、delete、select for update

##### ③ select不加锁

#### (2) 间隙锁

锁定索引记录间隙，innoDB使用next-=确保间隙不变，防止insert产生幻读现象（明明没有数据行但却显示已插入数据），在RR隔离级别下支持

##### ①索引上的等值查询（唯一索引），给不存在的记录加锁时，优化为间隙锁

##### ②索引上的等值查询（普通索引），向右遍历时最后一个值不满足查询需求时，next-key lock退化为间隙锁（将之前和之后的间隙都锁住）（临键锁）

##### ③索引上的范围查询（唯一索引），会访问到不满足条件的第一个值为止（锁住范围内的所有数据）（临键锁）

##### ④间隙锁可以共存，两个事务可以在同一间隙使用间隙锁

#### (3) 临键锁

锁定索引记录间隙和数据（行锁和临键锁组合），在RR隔离级别下支持
